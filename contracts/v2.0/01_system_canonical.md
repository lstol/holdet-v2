# Holdet v2 — System Canonical
# 01_system_canonical.md

---

## 1. System Identity

Holdet v2 models cycling race outcomes as causally consistent probability distributions derived from rider capabilities and race conditions. It produces Expected Value estimates over a fixed payoff structure for decision support.

The system is:
- **mechanistic** — it models structure, not preferences
- **contracts-first** — layer boundaries are enforced before implementation
- **non-adaptive at runtime** — no learning or feedback occurs during execution

---

## 2. Global Definitions

All terms used across all files are defined here. Layer files use these definitions without modification.

---

### Dependency

Component A depends on component B if:
- A reads values derived from B
- A uses parameters trained on data derived from B
- A shares mutable state with B

"Derived from" includes direct use, aggregation, transformation, and parameter training influence — regardless of the number of intermediate steps. Indirect access via intermediate artifacts does not break dependency. All such paths are treated as direct dependencies.

---

### Raw Input

An external observation that:
- is not generated by this system or any prior run of this system
- is recorded independently of system computation
- is time-indexed and snapshotted before execution

Outputs from prior system runs are explicitly forbidden as raw inputs, including logs, EV outputs, probability outputs, or traces from any prior execution.

---

### Simulation

A process that generates outcome distributions through iterative or path-dependent state evolution.

Permitted use: model development and structural validation only, prior to the execution boundary.

Prohibited use: simulation outputs must not enter any layer at runtime. Simulation must not be the sole or dominant determinant of any layer parameter. Simulation findings must not guide, adjust, or select model parameters — including indirectly. Post-hoc evaluation of fixed parameters is the only permitted runtime-adjacent use.

All simulation artifacts must be finalized, versioned, and immutable before the execution boundary is declared.

---

### Context-Independent

A variable is context-independent if it can be computed without access to:
- race-specific inputs, past or present
- stage-specific inputs, past or present
- time-indexed race data within the exclusion window

Historical data indexed or conditioned on specific races or stages is prohibited from Layer 0 regardless of how far outside the exclusion window it falls.

---

### Execution Boundary

The declared transition point from S1 (Context Freeze) to S2 (Model Freeze) in the execution state machine.

It is the moment at which all inputs are confirmed frozen and before parameters are loaded. It is not inferred retrospectively. It is not the start of computation.

Authority: must be declared by a process independent of parameter training, simulation, and evaluation systems. The declaring process must not be co-located with training infrastructure and must produce an externally verifiable record.

Enforcement: recorded in the Governance Binding Certificate before S2 begins. Any parameter updated after the boundary record is invalid for that run. The boundary cannot be redefined retroactively.

---

### Exclusion Window

A globally defined, versioned duration applied uniformly across all riders and all runs.

- Must be greater than zero
- Must exceed the total duration of the current race (not a single stage)
- Must be consistent across all riders and runs within a version
- Cannot be user-defined or runtime-adjustable
- Changes require Governance Layer approval

No data whose timestamp falls within the exclusion window prior to race start may enter Layer 0 — directly, indirectly, or through aggregation. Any aggregation whose input domain intersects the exclusion window is prohibited regardless of weighting.

---

### Outcome Space

The complete, closed set of atomic observable race and financial events over which probability distributions are defined and EV is computed. Defined in full in 03_outcome_space.md. No event outside that set may be modeled.

---

## 3. System Architecture

### Layer Structure

| ID | Name | Role |
|----|------|------|
| 0 | Rider-Intrinsic | Stable, race-independent rider properties |
| 1 | Race-State | Dynamic, time-indexed race context |
| 2 | Interaction | Derived features combining Layer 0 and Layer 1 |
| 3 | Probability | Outcome distributions over Outcome Space |
| 4 | EV | Expected value from probabilities and payoff |
| G | Governance | Configuration authority and evolution control |
| E | Execution | State machine governing computation order |
| SIV | System Integrity Validator | Pre-execution structural enforcement |

### Permitted Data Flows

```
Layer 0 ──────────────────────► Layer 2
Layer 1 ──────────────────────► Layer 2
Layer 2 ──────────────────────► Layer 3
Layer 3 ──────────────────────► Layer 4
Payoff Structure (RULES.md) ──► Layer 4
Outcome Space ────────────────► Layer 3
Outcome Space ────────────────► Layer 4
```

### Forbidden Flows (Complete)

- Layer 4 → any lower layer
- Layer 3 → any lower layer
- Layer 2 → Layer 1 or Layer 0
- Layer 1 → Layer 0
- EV → Payoff Structure
- EV → Outcome Space
- EV → Parameter Training (current or future run)
- Prior run outputs → current run inputs (unless declared persistent state via Governance)
- Simulation outputs → any runtime layer
- External proxy signals (market odds, rankings, model outputs) → Layer 0, Layer 3, or Payoff Structure

### System Boundary

The system boundary encompasses Layers 0–4, the Execution Protocol, the SIV, and the Governance Layer. Payoff structure and execution boundary declaration are external inputs governed by this specification. Processes outside the boundary are subject to 06_governance.md and may not re-enter the system through unconstrained channels.

---

## 4. Objective Separation

**System Objective:** Model cycling race outcomes as causally consistent probability distributions. Strictly mechanistic. Does not encode competitive ranking goals. Optimizes for structural correctness of probability distributions and EV consistency. No external performance metric may influence system behavior.

**Evaluation Objective:** How system outputs are assessed externally (competition ranking, predictive accuracy, comparative performance). Strictly post-hoc. Evaluation results must not enter the system as inputs or influence probability modeling, EV computation, or parameter selection — in the current run or through any cross-run path.

**Human Objective:** User or operator goals external to the system (e.g. achieve top Holdet ranking). Not a system input. Must not influence any internal system layer. May guide external strategy decisions outside the system boundary only. Must not inform payoff structure selection or execution boundary timing.

**Separation invariants:**
1. System Objective is independent of Evaluation Objective
2. Evaluation Objective is independent of system computation
3. Human Objective is external to both
4. No feedback loops are permitted between objective layers

## 5. Global Invariants

Single authoritative list. Per-layer restatements of these invariants are subordinate to this section.

**GI-1 — No Downward Mutation**
No layer may modify any lower-numbered layer. Applies to all layers without exception, including Layers 3 and 4 writing to Layer 1.

**GI-2 — No Upward Dependency Violation**
Higher layers may depend on lower layers only through the permitted flows in Section 3.

**GI-3 — Reproducibility**
All outputs must be reproducible from declared inputs and fixed parameters. All Race-State computations must reference a fixed snapshot of raw inputs. Live data sources must be snapshotted before the execution boundary.

**GI-4 — Layer Origin Declaration**
All variables must declare their layer of origin. Undeclared origin is a validation failure.

**GI-5 — State Isolation**
No mutable state may be shared across layers. All inter-layer communication must occur through explicit, immutable outputs within the execution state machine.

**GI-6 — Execution Boundary Time-Ordering**
All parameter training, simulation, and configuration changes must occur before the execution boundary is declared. No modification to inputs, parameters, or configurations is allowed after the boundary is sealed.

**GI-7 — Cross-Run Isolation**
No execution state or output from run N may influence run N+1 inputs, parameters, or configuration, unless explicitly declared as persistent state through a Governance Layer mechanism. This applies indefinitely across all runs.

**GI-8 — Feedback Prohibition**
No feedback loop is permitted between any layers in any direction — including through logging, external analysis, parameter retraining informed by outputs, payoff structure updates informed by EV, or human interpretation chains that re-enter the system.

**GI-9 — Violation Invalidation**
Any constraint violation invalidates the computation in which it occurs. Outputs from an invalidated run must be discarded and must not enter any downstream process.

**GI-10 — Governance Binding**
Every execution run must be bound to exactly one valid Governance Binding Certificate. Execution without a valid GBC is structurally invalid.
