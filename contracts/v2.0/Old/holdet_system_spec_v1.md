> Single authoritative reference. Supersedes all individual contract files.

## PART I — GLOBAL DEFINITIONS

All terms used in this specification are defined once here. All layers use these definitions without modification.

---

### 1.1 Dependency

Component A **depends on** component B if:
- A reads values derived from B
- A uses parameters trained on data derived from B
- A shares mutable state with B

**"Derived from" includes:** direct use, aggregation, transformation, and parameter training influence — regardless of the number of intermediate steps.

**Indirect access via intermediate artifacts does not break dependency.** A layer that consumes an artifact encoding, aggregating, or transforming another layer's outputs is considered directly dependent on that layer.

---

### 1.2 Raw Input

A raw input is an external observation that:
- is not generated by this system or any prior run of this system
- is recorded independently of system computation
- is time-indexed and snapshotted at a fixed point before execution

**Constraint:** Outputs from prior system runs are explicitly forbidden as raw inputs, including logs, traces, EV outputs, or probability outputs from any prior execution.

---

### 1.3 Simulation

A process that generates outcome distributions through iterative or path-dependent state evolution.

**Permitted use:** Simulation may only be used prior to the execution boundary for model development, structural validation, and parameter grounding verification.

**Prohibited use:** Simulation outputs must not be used as runtime inputs to any layer. Simulation must not be the sole or dominant determinant of any layer parameter. Simulation findings must not be used to guide, adjust, or select model parameters — even indirectly. Simulation may be used for post-hoc evaluation of fixed parameters only.

**Artifacts:** Any artifact produced by simulation must be finalized, versioned, and immutable before the execution boundary is declared.

---

### 1.4 Context-Independent

A variable is context-independent if it can be computed without access to:
- race-specific inputs, past or present
- stage-specific inputs, past or present
- time-indexed race data within the exclusion window

**Constraint:** Historical data that is indexed or conditioned on specific races or stages is prohibited from Layer 0, regardless of how far outside the exclusion window it falls.

---

### 1.5 Outcome Space

The complete, closed set of atomic observable race and financial events over which probability distributions are defined and EV is computed. Defined in full in Part V. No event outside this set may be modeled.

---

### 1.6 Execution Boundary

**Definition:** The execution boundary is the declared transition point from S1 (Context Freeze) to S2 (Model Freeze), as defined in the Execution State Machine (Part VII).

**Precise meaning:** The boundary is the moment at which all inputs and parameters are confirmed frozen. It is not inferred retrospectively. It is not the point at which computation begins — it is the point at which the configuration record is sealed.

**Authority:** The execution boundary must be declared by a process independent of parameter training systems, simulation systems, and evaluation systems. The declaring process must not be co-located with training infrastructure.

**Enforcement:** The execution boundary must be recorded in the Governance Binding Certificate (GBC) before S2 begins. Any parameter updated after the boundary record is invalid for that run. The boundary cannot be redefined retroactively.

---

### 1.7 Exclusion Window

A globally defined, versioned duration applied uniformly across all riders and all runs.

**Constraints:**
- Must be greater than zero
- Must exceed the total duration of the current race (not a single stage)
- Must be consistent across all riders and all runs within a version
- Cannot be user-defined or runtime-adjustable
- Changes require Governance Layer approval

**Effect:** No data whose timestamp falls within the exclusion window prior to race start may enter Layer 0, directly, indirectly, or through aggregation. Any aggregation whose input domain intersects the exclusion window is prohibited regardless of weighting.

---

### 1.8 Execution Boundary Independence

A process is independent of parameter training for the purpose of boundary declaration if it operates under separate authority, cannot be triggered by training completion, and produces a verifiable external record.

## PART II — SYSTEM ARCHITECTURE

---

### 2.1 Layer Structure

| Layer | Name | Role |
|---|---|---|
| 0 | Rider-Intrinsic | Stable, race-independent rider properties |
| 1 | Race-State | Dynamic, time-indexed race context |
| 2 | Interaction | Derived features combining Layer 0 and Layer 1 |
| 3 | Probability | Outcome distributions over Outcome Space |
| 4 | EV | Expected value from probabilities and payoff |
| G | Governance | Configuration authority and evolution control |
| E | Execution | State machine governing computation order |
| SIV | System Integrity Validator | Pre-execution structural enforcement |

---

### 2.2 Permitted Data Flows

```
Layer 0 ──────────────────────────────────► Layer 2
Layer 1 ──────────────────────────────────► Layer 2
Layer 2 ──────────────────────────────────► Layer 3
Layer 3 ──────────────────────────────────► Layer 4
RULES.md (Payoff Structure) ──────────────► Layer 4
Outcome Space ────────────────────────────► Layer 3
Outcome Space ────────────────────────────► Layer 4
```

---

### 2.3 Forbidden Flows (Complete List)

The following flows are forbidden without exception:

- Layer 4 → any layer below Layer 4
- Layer 3 → any layer below Layer 3
- Layer 2 → Layer 1
- Layer 2 → Layer 0
- Layer 1 → Layer 0
- EV → Payoff Structure
- EV → Outcome Space
- EV → Parameter Training
- Probability → Parameter Training (where parameters are EV-derived)
- Any layer → Rider-Intrinsic State (post-initialization)
- Prior run outputs → current run inputs (unless explicitly declared persistent state via Governance)
- Simulation outputs → runtime layer inputs
- External proxy signals (market odds, rankings, model outputs) → Layer 0, Layer 3, or Payoff Structure

---

### 2.4 System Boundary

The system boundary encompasses Layers 0–4, the Execution Protocol, the System Integrity Validator, and the Governance Layer. Payoff structure and the execution boundary declaration process are external to the system but are governed by this specification. Processes operating outside the system boundary are subject to the Decision Governance Protocol and may not re-enter the system through unconstrained channels.

## PART III — LAYER DEFINITIONS

---

### Layer 0 — Rider-Intrinsic

**Definition:** The set of rider properties that describe a rider independently of any specific race event. Properties are stable or slowly evolving. This layer is strictly descriptive and has no evaluative or predictive authority.

**Permitted attributes:**

*Physiological Capacity* — Sustained power output capacity, anaerobic capacity, fatigue resistance profile. Must be derived from non-race-outcome data sources or externally measured physiological data. Must not be inferred from race timing, gaps, splits, positional deltas, or finishing positions through any number of transformation steps.

*Terrain Affinity Profile* — Climbing suitability, sprint suitability, time-trial suitability, mixed terrain suitability. Must be computed independently of race outcome data. Must not encode ranking or finishing position information.

*Consistency Profile* — Performance variance across comparable efforts, stability of output under repeated conditions. Must be computed only from non-race-derived performance measurements. Any metric derived from race timing, gaps, splits, or positional changes is classified as race-derived and prohibited.

*Recovery Dynamics* — Fatigue accumulation rate, recovery rate between efforts, multi-stage endurance characteristics. Must be derived only from training load data, physiological measurements, or controlled non-competitive performance tests. A non-competitive performance test is a controlled measurement event that is not conducted under sanctioned race conditions, does not produce published results, and is not classified as a competitive event by any governing body. Multi-stage race data is classified as race-derived and is prohibited. If sufficient non-race data is unavailable, the attribute must be treated as unobserved rather than inferred.

**Freeze constraint:** Rider-Intrinsic values must be frozen at race initialization. Values cannot be recomputed, updated, or replaced during race execution. Any change constitutes a contract violation.

**Provenance constraint:** All Rider-Intrinsic inputs must originate from data whose timestamp falls outside the exclusion window. No data from any part of the current race may be included. Any aggregation whose input domain intersects the exclusion window is prohibited regardless of weighting.

**Time horizon constraint:** Rider-Intrinsic updates must use aggregation windows strictly greater than the exclusion window plus a fixed safety margin defined at system initialization. The safety margin value must be declared in the Governance Binding Certificate. The safety margin must be greater than zero.

**Information source constraint:** Rider-Intrinsic attributes may not be derived from any transformation of race data, regardless of representation, intermediate variables, or processing steps. Any data originating from race events — including splits, pacing, gaps, positional deltas — is classified as race-derived and prohibited, even if transformed, normalized, aggregated, or not explicitly labeled as race outcome data.

**Prohibited constructs:** Race finishing positions, leaderboard rankings, stage results of any kind, evaluation metrics, simulation-derived performance outputs, external proxy signals including betting markets, rankings, and odds.

**Ontological separation invariant:** This layer must remain independent of the Evaluation Objective, System Objective, Payoff Structure, simulation outputs, and all external proxy signals. No transformation of external or evaluation data may enter Rider-Intrinsic State under any representation.

---

### Layer 1 — Race-State

**Definition:** The set of variables describing the current race environment and its evolution over time. All variables are tied to a specific race event or stage and do not persist beyond that scope.

**Permitted components:**

*Course Structure* — Elevation profile, distance, segment classification (climb, flat, descent), technical characteristics. Must be derived from course design data, not rider outcomes.

*Environmental Conditions* — Weather conditions (wind, temperature, precipitation), road surface conditions. Must be exogenous to rider performance. Must not be inferred from race outcomes.

*Race Progression State* — Current race phase (early, mid, final), breakaway presence and structure, peloton composition and fragmentation, time gaps between groups. May be derived from race telemetry and observation. Must not be stored or reused outside the current race context.

**Constraint on progression state:** Peloton composition and time gaps encode cumulative performance differentials. These variables are permitted as race observations. Downstream layers must not use them to reconstruct GC standings or evaluation signals.

*Temporal Position* — Elapsed distance, remaining distance, time since race start. Must be strictly tied to current race instance. Must not reference historical race timelines.

**Isolation constraint:** Race-State must remain fully isolated from Rider-Intrinsic State. Race-State data must not be written to or used to update Rider-Intrinsic attributes. No aggregation of Race-State data may be persisted beyond the race event. Derived outputs must be discarded after race completion. This constraint binds all consuming layers, not only Layer 2.

**Read-only constraint:** Layer 1 is read-only for all consuming layers. No downstream layer — including Layers 2, 3, and 4 — may modify Race-State.

**Prohibited constructs:** Evaluation metrics, EV outputs or derived signals, simulation outputs used as runtime inputs, data originating from Rider-Intrinsic updates.

---

### Layer 2 — Interaction

**Definition:** Derived variables representing how rider-intrinsic characteristics interact with current race conditions. This layer is purely transformational. It has no evaluative or predictive authority and does not define probabilities, EV, or outcomes.

**Permitted inputs:** Layer 0 and Layer 1 only. No external data sources are permitted.

**Output constraint:** All Interaction Outputs must be computable solely from Rider-Intrinsic and Race-State inputs and contain no hidden state, learned parameters, or opaque transformations.

**Prohibited output constructs:** Probability estimates, expected value calculations, ranking or scoring systems, any variable that is individually equivalent to predicted finishing position or rank, any construction of a proxy for prohibited variables.

**Ranking prohibition (joint):** Interaction Outputs must not produce an orderable ranking of riders when computed across the rider set, even if no individual output is equivalent to a rank. Any output computable as a direct ordering of riders over a race field constitutes a ranking and is prohibited.

**Expectation prohibition:** Outputs labeled as "expectations," "expected performance," or similar must not encode probability-weighted outcomes. Interaction Outputs represent capability-condition matching only, not likelihood-weighted results.

**No state mutation:** This layer must not modify Rider-Intrinsic State or Race-State. It must not persist any derived data beyond execution scope.

**No temporal feedback:** Interaction Outputs must not be used to update Rider-Intrinsic attributes. Interaction Outputs must not be reused across race events. No cross-race aggregation is permitted.

**Isolation constraint:** Interaction Outputs must not be stored, cached, or reused beyond the current computation. No artifact containing Interaction Outputs may persist across runs.

**Input provenance:** This layer has no mechanism to validate the provenance of its inputs. Provenance compliance for Layer 0 and Layer 1 is enforced by the System Integrity Validator before execution begins.

---

### Layer 3 — Probability

**Definition:** The only layer permitted to define probability distributions over race outcomes. All probabilistic modeling occurs exclusively within this layer. This layer does not compute expected value.

**Permitted inputs:** Interaction Layer outputs (Layer 2) only. No direct dependency on Layer 0 or Layer 1. No dependency on Layer 4. No dependency on Evaluation or Human Objective layers.

**Output constraint:** All outputs must be valid probability distributions over the defined Outcome Space. Probabilities must sum to 1 where applicable. All probabilities must be explicitly represented — no implicit or embedded probabilities are permitted.

**Structural independence from EV:** No EV values, signals, or transformations may be used in probability computation. No parameter, weight, or model component may be derived from EV outputs. Probability distributions must be definable without reference to payoff structure.

**Parameter provenance:** Parameters must not be derived from EV outputs, evaluation metrics, or simulation outputs. Parameters must be fixed prior to the execution boundary. Parameters must be versioned and reproducible.

**Outcome space constraint:** The outcome space modeled by this layer must be selected independently of payoff structure. The set of outcomes for which probabilities are computed must not be determined by which outcomes are rewarded under RULES.md. Outcome space selection is governed by the Governance Layer and must be declared in the Governance Binding Certificate.

**Simulation constraint:** Layer 3 parameters must not be derived from simulation outputs. Simulation may be used for post-hoc validation of fixed parameters only. Simulation outputs must not be used as inputs, features, or parameter sources.

**Computation constraint:** Probability outputs must be computable from Interaction Outputs and fixed parameters only. No simulation process may be invoked during execution. No stochastic sampling is permitted at runtime.

**Inspectability:** The mapping from inputs to probabilities must be transparent, explicit, and reproducible. No hidden or opaque transformations are permitted.

**Isolation:** Probability outputs must not be written back to any upstream layer. No shared mutable state is permitted between this layer and any other layer.

---

### Layer 4 — EV (Expected Value)

**Definition:** The only layer permitted to compute expected value. EV is a descriptive metric, not a control signal.

**Formula:** EV = Σ(P(outcome) × payoff(outcome)) over all outcomes in Outcome Space.

**Permitted inputs:** Probability Layer outputs (Layer 3) and Payoff Structure only. No dependency on Interaction Layer, Rider-Intrinsic Layer, Race-State Layer, Evaluation Layer, or Human Objective Layer.

**Rider-Intrinsic independence:** EV must not depend on Rider-Intrinsic inputs through any path. All Rider-Intrinsic influence must be mediated exclusively through Layer 3 probabilities.

**Directionality:** Information flow into EV is strictly unidirectional. EV must not influence probability modeling, parameters, or feature construction. EV must not be used as input to any upstream layer.

**Output constraints:** EV outputs must not be stored as inputs to any upstream layer. EV outputs must not modify Probability Layer parameters. EV outputs must not be used in training, calibration, or feature generation in the current run or any future run.

**Cross-run constraint:** EV outputs logged for any purpose become available at training time. Any use of logged EV outputs in subsequent parameter training constitutes a dependency violation. This constraint spans runs.

**Computation:** EV must be computed deterministically from fixed probabilities and fixed payoff structure. No simulation, sampling, or stochastic estimation is permitted. No runtime learning or adaptation is permitted.

**Feedback suppression:** No information derived from EV may propagate backward into any layer below EV, including through logging, caching, analyst interpretation, or governance processes.

---

### Payoff Structure

**Definition:** A static, exogenous mapping from Outcome Space events to value, defined outside the system. Source: RULES.md.

**Constraints:**
- Must not depend on Rider-Intrinsic, Race-State, Interaction outputs, Probability outputs, or EV outputs
- Must be versioned and fixed prior to execution boundary declaration
- Must remain invariant across all runs within a version
- Must not be updated using EV outputs, derived signals, or any analysis derived from system outputs, including human judgment informed by system behavior
- Must not be correlated with EV through external market signals used as a basis for updates

**Versioning:** Changes to payoff structure require Governance Layer approval. The approving agent must not also control the Outcome Space definition. Change justification must not reference EV outputs.

**Cross-run stability:** Payoff structure must remain identical across runs sharing the same GBC version. Between GBC versions, changes must be approved through the Decision Governance Protocol.

---

### Objective Layer

**System Objective (internal):** To model cycling race outcomes as causally consistent probability distributions derived from rider capabilities and race conditions. This objective is strictly mechanistic and does not encode competitive ranking goals. The system optimizes for structural correctness of probability distributions and EV consistency. No external performance metric is permitted to influence system behavior.

**Evaluation Objective (external):** Defines how system outputs are assessed externally (e.g., competition ranking, predictive accuracy, comparative performance). Evaluation is strictly post-hoc. Evaluation results must not be used as inputs to the system. No evaluation metric may influence probability modeling, EV computation, or parameter selection, in the current run or through any cross-run path.

**Human Objective (external):** Represents user or operator goals external to the system (e.g., achieve top ranking in Holdet competition). Human objectives are not system inputs. Human objectives must not influence any internal system layer. Human objectives may only guide external strategy decisions outside the defined system boundary. Human objectives must not inform payoff structure selection or execution boundary timing.

**Separation invariants:**
1. System Objective is independent of Evaluation Objective
2. Evaluation Objective is independent of system computation
3. Human Objective is external to both system and evaluation
4. No feedback loops are permitted between objective layers

**Operationalization constraint:** "Structural correctness" and "EV consistency" as optimization targets must be defined and measured without reference to competition rankings, evaluation outcomes, or EV maximization. Definitions must be declared in the Governance Binding Certificate and must not be updated based on system performance signals.

## PART IV — GLOBAL INVARIANTS

Single authoritative list. All prior per-layer statements of these invariants are superseded by this section.

---

### GI-1 — No Downward Mutation

No layer may modify any lower-numbered layer. This applies to all layers without exception, including Layers 3 and 4 writing to Layer 1.

---

### GI-2 — No Upward Dependency Violation

Higher layers may depend on lower layers only through the explicitly permitted flows defined in Section 2.2. All other dependencies are forbidden.

---

### GI-3 — Reproducibility

All outputs must be reproducible from their declared inputs and fixed parameters. All Race-State computations must reference a fixed snapshot of raw inputs. Live or streaming data sources must be versioned or snapshotted before the execution boundary.

---

### GI-4 — Layer Origin Declaration

All variables must declare their layer of origin. Undeclared origin is a validation failure.

---

### GI-5 — State Isolation

No mutable state may be shared across layers. All inter-layer communication must occur through explicit, immutable outputs produced within the Execution State Machine.

---

### GI-6 — Execution Boundary Time-Ordering

All parameter training, simulation, and configuration changes must occur before the execution boundary is declared. No modification to inputs, parameters, or configurations is allowed after the boundary is sealed. The boundary cannot be redefined retroactively.

---

### GI-7 — Cross-Run Isolation

No execution state or output from run N may influence run N+1 inputs, parameters, or configuration, unless explicitly declared as persistent state through a Governance Layer mechanism. This constraint applies across all runs indefinitely, not only between adjacent runs.

---

### GI-8 — Feedback Prohibition

No feedback loop is permitted between any layers in any direction. This includes indirect feedback through logging, external analysis, parameter retraining informed by outputs, payoff structure updates informed by EV, and human interpretation chains that re-enter the system.

---

### GI-9 — Violation Invalidation

Any violation of a constraint in this specification invalidates the computation in which it occurs. Outputs from an invalidated run must be discarded and must not enter any downstream process as data.

---

### GI-10 — Governance Binding

Every execution run must be bound to exactly one valid Governance Binding Certificate. Execution without a valid GBC is structurally invalid.

## PART V — OUTCOME SPACE

The outcome space is a closed set. Only events defined here exist for the system. No derived or composite event may exist unless explicitly defined below.

**Core principle:** Outcomes are atomic, observable race events or state transitions. No outcome encodes value, desirability, EV, or ranking.

**Stage Outcomes:**
- StageWin(rider, stage_t)
- StageFinishPosition(rider, position, stage_t) — position ∈ ℕ
- StageTopK(rider, k, stage_t)

**GC Outcomes:**
- GCPosition(rider, position, stage_t) — recomputed after each stage; derived race-state observable, not a preference signal

**Sprint / KOM Micro-Events:**
- SprintPoint(rider, sprint_id, stage_t)
- KOMPoint(rider, climb_id, stage_t)

**Jersey State Events (terminal per stage):**
- JerseyHold(rider, jersey_type, stage_t) — jersey_type ∈ {YELLOW, GREEN, KOM, WHITE, MOST_AGGRESSIVE}; evaluated only at stage completion

**Team Events:**
- TeamStageWin(team, stage_t)
- TeamPodium(team, stage_t, position ∈ {1,2,3})

**Race Completion Events:**
- FinishStatus(rider, stage_t) ∈ {FINISH, DNF, DNS, DSQ}

**Transfer Events (financial state space):**
- TransferBuy(rider, price, time_t)
- TransferSell(rider, price, time_t)
- TransferFeeApplied(rider, fee_amount, time_t)

Transfer events are not race outcomes. They belong to financial state space and are represented for EV closure. TransferFeeApplied is deterministic: fee = 0.01 × TransferBuy.price. No probabilistic interpretation is permitted.

**Anti-compression constraint:** No function may map multiple Outcome events into a single latent scalar unless explicitly defined in this specification.

**Closure constraint:** Any observable race or financial event not in this Outcome Space is undefined and must not be modeled.

**Governance constraint:** The Outcome Space may only be modified through Governance Layer approval. The approving agent must not also control Payoff Structure.

## PART VI — PROVENANCE MODEL

---

### 6.1 Variable Origin

Every variable entering any layer must have a declared origin satisfying one of:
- External raw input (as defined in Section 1.2)
- Derived from declared Layer 0 attributes within provenance constraints
- Derived from Race-State telemetry or course design data
- Derived from a preceding layer's outputs through a permitted flow

---

### 6.2 Contamination Prevention

**At Layer 0:** Provenance is enforced by the exclusion window and the Information Source Constraint. The System Integrity Validator checks for EV-correlated proxies in Layer 0 before execution.

**At Layer 2:** Input provenance is validated by the SIV before execution. Layer 2 itself cannot validate provenance of its inputs at runtime.

**At Layer 3:** Parameter provenance is declared and versioned in the GBC. The SIV validates parameter lineage before execution.

**At Layer 4:** Payoff structure provenance is declared and versioned. EV output contamination of any upstream layer is blocked by GI-8 and enforced by the SIV cross-run contamination check.

---

### 6.3 Cross-Run Influence

Cross-run influence is permitted only through Governance Layer mechanisms. A persistent state artifact must be: explicitly declared as persistent, versioned, approved under the Decision Governance Protocol, and not derived from EV outputs, evaluation metrics, or simulation findings. All other cross-run paths are forbidden under GI-7.

---

## PART VII — EXECUTION PROTOCOL

---

### 7.1 State Machine

Execution is a finite state machine with strictly ordered states. No cross-state mutation is permitted.

**S0 — INITIALIZATION**
- Load RULES.md (immutable)
- Load Outcome Space (immutable)
- Load system configuration
- Verify Governance Binding Certificate
- Initialize dependency graph
- No computation permitted; no probabilistic or model activity permitted

**S1 — CONTEXT FREEZE**
- Load Rider-Intrinsic Layer (Layer 0) snapshot
- Load Race-State Layer (Layer 1) snapshot
- Freeze all upstream inputs
- Inputs become immutable; any modification after this point invalidates execution

**EXECUTION BOUNDARY** — declared here, recorded in GBC, immutable from this point

**S2 — MODEL FREEZE**
- Load Interaction Layer parameters
- Load Probability model parameters
- Load EV model parameters
- All parameters become immutable; no retraining, recalibration, or adjustment permitted

**S3 — INFERENCE EXECUTION**
- Compute Interaction Outputs (Layer 2)
- Compute Probability distributions (Layer 3)
- Compute Expected Value (Layer 4)
- No new data sources permitted; no structural modification permitted; no external input permitted
- System Integrity Validator runs after each layer

**S4 — OUTPUT FINALIZATION**
- Compute final EV-ranked outputs
- Emit system outputs
- Freeze output artifact
- Outputs become immutable upon emission; no feedback into any prior layer permitted

---

### 7.2 Failure Handling

Any constraint violation detected during execution results in:
- Immediate classification of execution as INVALID
- Discard of all outputs from current run
- Structured violation report (violated rule ID, affected nodes, dependency path trace)
- Forwarding to Simulation Harness for analysis
- No attempt at correction by any executing process

---

### 7.3 Post-Execution Cleanup

- All temporary state is destroyed
- No runtime memory persists
- Only versioned outputs may be stored externally
- Interaction Outputs are discarded unless explicitly versioned and declared through Governance mechanisms

---

## PART VIII — GOVERNANCE LAYER

---

### 8.1 Scope

The Governance Layer governs configuration authority for all system-critical parameters. It does not participate in execution runs and does not compute outcomes, probabilities, or EV. It governs: Outcome Space Definition, Payoff Structure, Exclusion Window, Parameter Training Regime, Execution Boundary Declaration, System Integrity Validator configuration.

---

### 8.2 Governance Binding Certificate (GBC)

Every execution run must reference exactly one GBC. A GBC is a signed configuration object containing:
- Version of Outcome Space
- Version of Payoff Structure (RULES.md)
- Version of Exclusion Window (including safety margin value)
- Version of Parameter Training Regime
- Version of Execution Protocol
- Version of System Integrity Validator
- Authority identifiers for each governance component
- Declared operationalization of System Objective targets

Once an execution run begins, its GBC cannot change. Any mismatch between runtime configuration and GBC invalidates execution.

---

### 8.3 Authority Separation

No single agent may have unilateral authority over more than one governance-critical component. Specifically prohibited combined authorities:
- Payoff Structure + Outcome Space
- Parameter Training Regime + Exclusion Window
- Execution Boundary declaration + Parameter Training
- Any governance role + EV outcome benefit

---

### 8.4 Change Requirements

Any structural change requires:
- Classification as: layer modification, parameter modification, structural constraint modification, or governance modification
- Approval from at least two independent governance perspectives
- Evidence spanning multiple execution runs
- No single-run EV, probability, or evaluation signal as primary justification

Prohibited as sole justification: single-run performance improvement, EV uplift in isolated runs, probability calibration gains without structural analysis.

---

### 8.5 Drift Detection

Governance must periodically evaluate system evolution for:
- Gradual convergence toward EV maximization over structural fidelity
- Implicit collapse of layer separation
- Unintended reintroduction of feedback loops
- Increasing reliance on cross-run feedback paths

Drift detection must be explicit and recorded. Detection is triggered by pattern analysis across multiple runs, not by individual run signals.

---

### 8.6 External Influence Constraint

External signals must not directly determine governance decisions. External signals may only inform governance decisions if they are explicitly declared, independent of system outputs, and not correlated with EV or evaluation targets.

---

## PART IX — SYSTEM INTEGRITY VALIDATOR

The SIV performs deterministic graph validation before execution. It blocks execution if any validation fails. It performs no probabilistic reasoning or semantic interpretation.

**Validation graph:** G = (Nodes, Edges) where Nodes = layers, variables, outcomes, parameters, rules; Edges = permitted dependencies. Only explicit edges are valid.

**Checks performed:**

*Outcome Coverage:* Every scoring element in RULES.md must map to ≥1 Outcome Space event.

*Outcome Isolation:* Outcome nodes must not contain EV values, ranking signals, or payoff information.

*Dependency Direction:* Only permitted flows (Section 2.2) may exist. Reverse edges and skipped-layer edges fail validation.

*Cross-Layer Leakage:* No upstream layer may encode downstream objectives. EV-correlated proxies in Layer 0 or 1, ranking signals in Layer 2, payoff-encoded features upstream of EV — all fail validation.

*Transfer Integrity:* TransferFeeApplied = 0.01 × TransferBuy.price, strictly deterministic.

*Temporal Boundary:* Execution boundary must be defined before S2, immutable after S2 begins, not inferred post-hoc.

*Cross-Run Contamination:* No outputs from run N may appear in run N+1 inputs unless declared as persistent state through Governance.

*GBC Completeness:* GBC must exist, be complete, and be version-consistent across all governance components.

**Outputs:** PASS (execution proceeds) or FAIL (execution blocked, with violated rule ID, affected nodes, dependency path trace, and reason for invalidation).

---

## PART X — SIMULATION HARNESS

The Simulation Harness is a controlled testing environment for validating the architecture. It is not part of the decision system. It is a verification-only subsystem.

**Scope:** Simulates Layer 0–4 computation flow, Execution Protocol phases, SIV checks, and Governance constraints as pre-validated inputs.

**Inputs:** Must be frozen prior to simulation start. All Execution Protocol rules apply identically to production. SIV must run on every simulated execution.

**Mode:** Diagnostic. Does not halt on first violation. Records all structural violations. Categorizes violations by layer breach, cross-layer leakage, governance violation, and execution protocol violation.

**Outputs:** Full execution trace, detected structural violations, cross-layer dependency graph, detected feedback loops. No probabilistic performance evaluation. No EV-based optimization.

**Anti-leakage:** Simulation outputs must not be used as training data unless explicitly versioned and declared outside the system boundary through Governance Layer procedures. Simulation findings are strictly observational. Parameter updates based on simulation findings violate GI-8 and the Simulation constraint in Section 1.3.

---

## PART XI — OPERATOR PROTOCOL (Claude Code)

Claude Code operates as a deterministic executor of this specification. It is not a system designer and must not modify architecture unless explicitly instructed via Governance Layer approval.

**Role:** Deterministic executor of defined pipelines, validator of constraint compliance, reporter of structural violations.

**Non-ambiguation rule:** If any layer behavior, dependency rule, input provenance, execution order, or governance authority is unclear, Claude Code must stop execution and request clarification. It must not proceed with inferred behavior.

**Execution fidelity:** Execute layers in strict order. Enforce all stated constraints verbatim. Treat all "must" statements as hard constraints. Treat all undefined terms as blocking conditions.

**No creative resolution:** Claude Code must not resolve contradictions, choose between conflicting constraints, infer missing definitions, or harmonize incompatible rules. All conflicts are escalated, not resolved.

**Violation handling:**
1. Classify: layer violation, cross-layer leakage, governance violation, or execution protocol violation
2. Mark execution INVALID
3. Produce structured violation report
4. Do not attempt correction

**System hierarchy:** Claude Code is subordinate to Governance Layer, Execution Protocol, SIV, and layer contracts, in that order.

---

## PART XII — DECISION GOVERNANCE PROTOCOL

Governs how humans may interpret system outputs and decide on structural modifications. Applies after execution, not during.

**Separation of interpretation and modification:** Interpreting system outputs is unrestricted. Acting on interpretations to modify system structure is regulated. No structural change may be made solely on single-run performance signals.

**Evidence requirement:** System modifications must be based on aggregated evidence across multiple runs. Evidence must include cross-run consistency checks and pattern stability analysis. Isolated EV or probability shifts are insufficient.

**Feedback containment:** System outputs may inform human understanding but must not directly dictate redesign. Interpretation must pass through a deliberate abstraction step before influencing changes.

**Conflict of interpretation:** When multiple interpretations of outputs exist, preference must be given to structural explanations over performance explanations. No single metric may dominate decision-making.

**Stability preference:** The system must prefer structural stability over performance optimization unless persistent multi-run degradation is observed or structural violation is detected by the SIV.

## PART XIII — CONFLICT RESOLUTION LOG

This section documents all conflicts identified during the merge of source files and the resolution applied.

---

**CR-01 — Execution Boundary Definition Conflict**

*Source:* contract_manifest.md defined the execution boundary as "the point where all inputs, parameters, and configurations are fixed and computation starts." execution_protocol.md defined it as the "transition from S1 → S2."

*Conflict:* The manifest definition implies the boundary is the start of computation. The protocol definition places the boundary before parameters are loaded (S2), creating an ambiguity about whether the boundary precedes or coincides with model freeze.

*Resolution:* The boundary is defined as the transition point from S1 to S2 — after inputs are frozen and before parameters are loaded. This places it precisely between Context Freeze and Model Freeze, making it the earliest point at which the configuration record can be sealed. All content from contract_manifest.md's Execution Boundary Constraint is superseded by Part VII of this document.

---

**CR-02 — "Directly" Qualifier in EV Layer**

*Source:* An earlier version of ev.md stated "EV must not directly depend on Rider-Intrinsic inputs," with the word "directly" creating a carve-out not present in the Dependency definition.

*Conflict:* The Dependency definition covers both direct and indirect dependencies. "Directly" was narrower and inconsistent.

*Resolution:* The qualifier "directly" is removed. EV must not depend on Rider-Intrinsic inputs through any path. The mediation-through-Layer-3 requirement is absolute.

---

**CR-03 — Simulation Influence Constraint**

*Source:* Earlier versions permitted simulation as a "supplementary signal" with quantitative thresholds undefined. The latest rider_intrinsic.md and probability.md versions tightened this, but the interaction.md file retained the term "simulation" in prohibited constructs without a definition.

*Resolution:* Simulation is defined once in Section 1.3. Simulation outputs are prohibited from all runtime layers. Simulation may be used for post-hoc evaluation of fixed parameters only. The "supplementary signal" language is removed and does not appear in this specification.

---

**CR-04 — Layer 1 Read-Only Scope**

*Source:* The read-only constraint on Layer 1 appeared only in the Layer 2 section of contract_manifest.md and interaction.md. Layers 3 and 4 were not explicitly bound.

*Resolution:* The read-only constraint on Layer 1 is stated as a global constraint in GI-1 (No Downward Mutation). It applies to all layers. The layer-specific statement in Layer 2 is retained for clarity but does not limit the scope of the global rule.

---

**CR-05 — "Terrain-Adjusted Performance Expectations" as Probability Surrogate**

*Source:* interaction.md listed "terrain-adjusted performance expectations" as a permitted example of Interaction Output. An expectation is a probability-weighted outcome by definition, conflicting with Layer 2's prohibition on probability estimates.

*Resolution:* The term "expectations" is prohibited from Interaction Outputs. The Expectation Prohibition in Layer 2 (Part III) replaces this example. Interaction Outputs represent capability-condition matching only.

---

**CR-06 — Payoff Structure Cross-Run Invariance Scope**

*Source:* ev.md stated payoff structure must remain "invariant across model execution runs," which was interpreted as within-run only by one reading.

*Resolution:* Payoff structure must remain identical across all runs sharing the same GBC version. Cross-version changes require Governance approval. The within-run reading is superseded.

---

**CR-07 — Outcome Space Selection Governance**

*Source:* probability.md prohibited payoff-aligned outcome space selection but placed no governance constraint on who defines the outcome space or under what authority.

*Resolution:* Outcome space selection is declared a governance-critical component in Part VIII. It must be declared in the GBC. The approving authority must not also control payoff structure.

---

**CR-08 — Safety Margin Value Unspecified**

*Source:* rider_intrinsic.md required a safety margin greater than zero but did not specify where its value is declared or governed.

*Resolution:* The safety margin value must be declared in the Governance Binding Certificate (Part VIII, Section 8.2). It is a governance-critical configuration parameter.

---

**CR-09 — "Non-competitive performance test" Undefined**

*Source:* rider_intrinsic.md permitted "controlled non-competitive performance tests" as a source for Recovery Dynamics without defining what qualifies.

*Resolution:* Defined in Layer 0 (Part III): a non-competitive performance test is a controlled measurement event not conducted under sanctioned race conditions, not producing published results, and not classified as a competitive event by any governing body.

---

**CR-10 — Stage Results Outside Exclusion Window**

*Source:* rider_intrinsic.md Section 5 prohibited stage results only "within exclusion window," while the Information Source Constraint prohibited all race data regardless of timing. These were in direct conflict.

*Resolution:* The Information Source Constraint takes precedence. All stage results, regardless of timing relative to the exclusion window, are prohibited from Layer 0 under the context-independence definition (Section 1.4). Historical data indexed or conditioned on specific races or stages is prohibited regardless of recency.

---

**CR-11 — System Boundary Definition**

*Source:* The objective layer and ev.md permitted activity "outside the system boundary" without defining where the boundary lies. This created an ungoverned re-entry point.

*Resolution:* The system boundary is defined in Section 2.4. Processes outside the boundary are governed by the Decision Governance Protocol (Part XII) and the constraints therein. External processes cannot re-enter the system through unconstrained channels.
